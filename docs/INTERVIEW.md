# 面试讲述脚本

> 配合简历项目「多语种车载意图路由器」使用。投 Momenta（自动驾驶 / 智能座舱方向）。
> 核心策略：**少讲技术实现，多讲产品判断与数据度量**。Momenta 的飞轮文化看重"用数据驱动迭代"，
> 这正是这个项目的主线。

---

## 一、90 秒电梯版（一面开场 / "讲讲你这个项目"）

> 用来快速给出全貌，让面试官决定往哪深挖。背熟到能脱口而出。

"这是我自己做的一个产品实践，针对车机多语种交互里一个被普遍忽略的场景——**混说**。

多语种市场的用户其实不会规规矩矩只说一种语言，他们会说'把 air conditioning 调到 20 度'，
中文里夹英文。但现有车机大多按单一语言训练，遇到这种就识别失败，用户被迫重说。
这在出海车型上直接影响语音功能的使用率。

我做了两件事：第一，把它定义成一个意图路由问题，用多语种大模型把任意语言、混说的指令
映射到固定的车控功能集；第二，也是我觉得更重要的，我建了一套**按语言维度切分的评测集**，
把'多语种交互到底好不好'变成可量化的指标——比如我能看到混说场景的准确率比单语种低多少。

我的判断是：模型层已经饱和，PM 的价值不在训模型，而在**定义场景、设计功能集、建立度量**。
所以这个项目我刻意把评测体系做成了一等公民。"

---

## 二、5 分钟展开版（STAR 结构）

### S — 背景 / 问题（30 秒）
- 多语种市场（含出海车型）用户日常 code-switching，一句话混用两种语言。
- 现有车机按单语训练 → 混说识别失败 → 用户重说 / 切语言 → 语音功能使用率受损。
- 行业现状：我扫了 GitHub 和商业方案，发现**没人**同时解决"混说 + 车载场景 + 可量化评测"。

### T — 我要解决什么（20 秒）
- 不是去训一个更强的模型，而是：①让混说指令能稳定落到车控功能；②让"做得好不好"可被度量、可按市场拆解。

### A — 我做的决策与执行（2.5 分钟，这是重点）
1. **竞品调研后定位差异点**：对比了 Rasa、wukong-robot、车机固定命令方案、模型底座等 5 类，
   做了对比矩阵，确认"混说 × 车载 × 评测"是空白，并把**混说意图准确率定为北极星指标**。
2. **设计产品定义层**：定义 8 类车控意图 + 槽位规范，做成可扩展结构——新增功能只改一处。
3. **建评测体系（我认为最有价值的部分）**：构造带标准答案的多语种评测集，
   分中文/英文/混说三个子集，做了一个按语言维度看准确率的质量看板。
4. **工程上做了双引擎降级**：云端 LLM 为主，本地规则兜底——因为车机对可用性要求极高，
   云端不可用时语音功能不能直接挂掉。
5. 全程输出了一份完整 PRD（竞品矩阵、需求优先级、指标体系、风险缓解、路线图）。

### R — 结果与洞察（40 秒）
- 跑通可演示的 demo，规则引擎基线就量化出了差距：单语种 100%、**混说 87.5%，gap 12.5pp**。
- 这个 gap 就是引入 LLM 路由要收敛的目标，也让"优化多语种交互"从拍脑袋变成了对着指标干。
- 更重要的收获是方法论：**任何交互质量问题，PM 的第一步应该是先建可拆解的度量，再谈优化。**

## 三、高频追问的标准答案

### Q1：为什么用 LLM 不用 Rasa 这类传统 NLU？
"Rasa 要按语言逐个标数据、训意图分类器，多语种和混说要嘛各训一套、要嘛覆盖不了。
LLM 的优势正是**零样本跨语言**，混说这种长尾场景不用专门标数据就能覆盖。
代价是时延和成本，所以我用规则引擎做兜底来对冲。这是个工程权衡，不是非黑即白。"

### Q2：评测集只有 27 条，够吗？
"现在这版是**种子集**，目的是先把评测框架和语言维度的拆解方式跑通、验证混说确实掉分。
真要上量产，这套集合要扩到每个目标市场几千条，按语种维护，并且要**从真实车机日志里回流**
高频和 bad case——这其实就接到了 Momenta 的数据飞轮思路：线上数据驱动评测集和模型一起迭代。"

### Q3（Momenta 几乎必问）：这个东西要上车量产，你会怎么做？
"几个关键点：
1. **可用性**：车机不能依赖云端常通，所以核心高频指令要有端侧模型或缓存，云端 LLM 处理长尾，
   我现在的双引擎降级就是这个思路的雏形。
2. **时延**：语音响应有硬性体验阈值，要设超时 + 降级，高频指令走本地快路径。
3. **评测闭环**：上车后从真实日志回流 bad case，按市场分语种持续扩评测集，做 A/B
   对比新旧方案在混说场景的使用率、重说率变化——用数据证明价值，而不是只看离线准确率。
4. **安全边界**：LLM 可能输出不存在的指令，所以路由层要做强校验，非法 intent 一律归兜底，
   绝不能让模型幻觉触发一个真实车控动作。"

### Q4：你怎么定义这个功能"做得好"？北极星指标为什么是它？
"北极星是**混说场景的意图准确率**，因为它直接对应差异化价值——单语种现有方案已经做得不错，
真正没被解决、又影响出海体验的是混说。我还会盯几个护栏指标：单语种别因为优化混说而退化、
兜底误触率（本该有效却判成 unknown）、以及降级可用率。"

### Q5：作为 PM，这个项目你自己写了代码，算 PM 的活吗？
"我写代码是为了**自己快速验证产品假设**，而不是要做工程。
真正体现 PM 价值的是：我对场景的定义（混说这个切入点）、功能集和优先级的设计、
以及把交互质量变成可度量指标这套方法。代码只是让我能不依赖资源就把想法跑出来、用数据说话。"

## 四、讲述时的注意事项

- **主动报数字**：87.5%、12.5pp gap、8 类意图、3 个语言子集——数字让叙述可信。
- **承认局限**：被问到 27 条样本、LLM 成本时，大方承认是种子版/有权衡，再讲怎么演进。
  PM 面试里"知道边界在哪"比"假装完美"加分。
- **往 Momenta 的飞轮和量产靠**：每次有机会就把话题引到"数据回流""线上迭代""量产可用性"。
- **别陷进技术细节**：面试官不关心 Streamlit 怎么写，关心你的产品判断。技术只在被追问时点到为止。
